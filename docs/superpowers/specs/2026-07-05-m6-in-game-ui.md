# Milestone 6 — In-Game UI & Path Styles Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Provide builders with an intuitive, aesthetically pleasing in-game UI to adjust path configurations dynamically. This allows builders to swap out palettes (e.g., base packed mud with mud brick variants, blending into coarse dirt and moss edges) and see the translucent preview update live, replacing manual JSON edits.

**Architecture:**
- `ui/PathStyleScreen`: A new Minecraft GUI screen (extending Forge's `Screen`) for editing path settings.
- `ui/components/`: Reusable, modern-looking UI widgets (visual block selectors, sliders for noise scales and blend widths).
- `config/PathStyle`: An expanded data structure representing a user's palette and edge choices, replacing or wrapping the static `OrganicTunables`.
- `tool/ToolManager`: Intercepts an input (e.g., a "Configure" button or hotkey) to open the `PathStyleScreen`.
- `plan/EditPlanBuilder`: Continues to receive the selected `PathStyle` and regenerates the `EditPlan` deterministically when the user changes a setting in the UI.

**Tech Stack:** Java 17, MinecraftForge 1.20.1 (Forge 47.4.10), Minecraft Client GUI framework.

## Global Constraints

- **Live Preview:** Adjusting sliders or changing blocks in the UI must immediately trigger a debounced rebuild of the `EditPlan` and update the `PreviewRenderer` behind the UI, without requiring the user to close the screen.
- **Premium Aesthetics:** The UI must look modern and match the mod's "Axiom-style" premium feel. Avoid raw text fields; use visual block icons, drag-and-drop or simple click-to-select for materials, and styled sliders.
- **Separation of Concerns:** The UI layer (`ui/`) must remain strictly separate from the pure computation (`organic/`, `plan/`). The UI only mutates a `PathStyle` object and requests a pipeline rebuild.

### Task 1: Data Model Expansion (`PathStyle`)

Expand the configuration system to support full, named path styles rather than just global numeric tunables. This matches the builder's workflow of having distinct layers (base, variants, outskirts).

**Files:**
- Create: `src/main/java/com/lwos/config/PathStyle.java`
- Create: `src/main/java/com/lwos/config/StyleManager.java`

**Requirements:**
- A `PathStyle` must define:
  - **Core Palette:** Base block (e.g., Packed Mud) and variant blocks (e.g., Mud Bricks) with associated weights and clustering noise scales.
  - **Edge Palette (Outskirts):** Blocks used for blending (e.g., Coarse Dirt, Moss Block).
  - **Blend Parameters:** Blend width (skirt size), erosion factors.
- `StyleManager` handles saving and loading these styles to JSON so users can reuse them across sessions.

- [ ] **Step 1: Define the `PathStyle` schema.** Create an immutable data class that holds the Core Palette, Edge Palette, and global parameters.
- [ ] **Step 2: Update `GradientEngine` and `BlendEngine`** to accept the new structured `PathStyle` rather than flat `OrganicTunables`. Ensure the compute side remains pure.
- [ ] **Step 3: Implement `StyleManager`** for JSON serialization/deserialization, allowing loading of default or custom styles.

### Task 4: Live Preview Integration

Ensure that the UI state directly drives the visual preview in real-time.

**Files:**
- Modify: `src/main/java/com/lwos/ui/PathStyleScreen.java`
- Modify: `src/main/java/com/lwos/tool/ToolManager.java`

**Requirements:**
- When a widget value changes (e.g., a slider is dragged), construct a new updated `PathStyle`.
- Trigger `ToolManager` to request a debounced rebuild of the `EditPlan`.
- `PreviewRenderer` will automatically catch the new `EditPlan` and update the 3D translucent mesh in the background.

- [ ] **Step 1: Wire the UI change listeners** to pass the updated `PathStyle` to the active tool.
- [ ] **Step 2: Implement debouncing** (e.g., 50-100ms) on the rebuild trigger so dragging a slider doesn't lag the client.
- [ ] **Step 3: Test the workflow:** Start a path, open the UI, set the core block to Packed Mud with a Mud Brick variant, set the outskirts to Coarse Dirt and Moss, drag the cluster size slider, and verify the preview updates seamlessly behind the UI.

## Definition of Done (Milestone 6)
- The user can open an in-game UI while a path is active.
- The UI contains distinct sections for the main path palette and the edge blending palette.
- Users can visually select blocks and adjust their weights and cluster sizes using styled sliders.
- Any change in the UI immediately updates the 3D block preview in the background.
- The UI is intuitive and aesthetically premium, providing a massive UX improvement over JSON editing.
- Configurations can be saved as named styles and reloaded.
