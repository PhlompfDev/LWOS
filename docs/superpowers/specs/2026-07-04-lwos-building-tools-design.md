# LWOS — Builder's Tools Mod: Design Spec

- **Date:** 2026-07-04
- **Status:** Approved design — ready for implementation planning
- **Scope of this document:** Master architecture + cross-cutting principles for all three subsystems and six milestones. **Milestone 1 is specified build-ready.** Milestones 2–6 are specified at the design/interface level; each gets its own plan cycle later.

---

## 1. Vision & goal

A Minecraft building-tools mod that lets a builder draw paths, terrain, gradients, and organic detail interactively — Axiom-style — and get results that look **hand-built by a skilled builder, not generated by a mathematically perfect algorithm.**

Target workflow:

```
Alt+Scroll → pick tool → right-click to place points → spline appears →
holographic preview (green place / red remove / blue terrain) →
adjust width & gradient → commit. Nothing changes until confirmation.
```

This replaces the manual builder loop (flatten → replace grass → coarse dirt → gravel → rooted dirt → randomize → fix edges → slabs → stairs) with **draw → preview → adjust → commit.**

### The single strongest design goal

> Favor clustered randomness over uniform randomness, irregular edges over straight ones, terrain preservation over flattening, and weighted material transitions over hard borders — so output resembles a skilled builder's imperfect-on-purpose work.

Everything below serves that goal.

---

## 2. Target platform & deployment

- **Loader / version:** MinecraftForge, Minecraft **1.20.1**.
- **Deployment:** **Singleplayer-first, multiplayer-ready by design.** Ships for singleplayer / creative / LAN first. All world mutation is routed through a **server-authoritative seam** (an `EditRequest` boundary) so adding dedicated-multiplayer support later is an implementation behind an existing interface, not a redesign.
- **Primary audience:** creative-mode builders.

---

## 3. Core design principles

These are testable requirements, not aspirations.

1. **One deterministic `EditPlan` drives both preview and commit.**
   The full path→blocks computation is a *pure function* of `(control points, parameters, operation seed)`, producing one immutable `EditPlan`. The preview renders that plan; commit applies that exact plan. `preview ≡ commit` holds **by construction**, not by keeping two code paths in sync.

2. **Determinism.** Same inputs + same seed → byte-identical `EditPlan`. The operation seed is fixed when the operation starts and travels with it. Nudging a parameter re-runs the pipeline deterministically. No stage may read wall-clock, unseeded RNG, or mutable global state.

3. **Clustered, not uniform, randomness.** Material choice and edge shape are driven by **low-frequency noise fields** (Perlin + cellular/Worley), so coarse blocks appear in patches and edges wander in runs. Per-block `random()` is only ever a tie-breaker, never the primary driver. *This is the single thing most competing mods get wrong.*

4. **Preserve terrain by default.** The default terrain mode **follows** the surface and replaces only top blocks. Flatten / average / carve are opt-in. The tool never levels a hill unless explicitly asked.

5. **Weighted transitions over hard borders.** Palettes are `(block, weight, noiseScale, clusterSize, edgePreference)` tuples. The outer band always feathers into surrounding terrain — no grass→path cliff.

6. **Compute/apply separation (boundary rule).** `geometry/` and `organic/` never import Minecraft world or rendering classes. They operate on a small read-only `WorldView` interface and plain data, so the hard logic is unit-testable headlessly and portable to other loaders later.

## 4. Architecture

### 4.1 Compute side vs. apply/render side

The system splits into a **compute side** (pure, deterministic, no MC rendering/IO) and an **apply/render side** (touches world + screen). The compute side is the bulk of the engineering and is fully unit-testable without a running game.

### 4.2 Module layout

```
lwos/
├── tool/                      # Subsystem 1: Interaction/UI (apply/render side)
│   ├── ToolManager            # active tool, Alt+scroll wheel, dispatch
│   ├── ToolInput              # raw input → intent (place point, drag handle, confirm/cancel)
│   ├── ToolRenderer           # tool wheel, control-point gizmos, width handles, HUD
│   └── tools/                 # PathTool, LineTool, CircleTool, FillTool… (one state machine each)
│
├── geometry/                  # Subsystem 2: Path generation (PURE)
│   ├── PathNode               # position, rotation, width, height offset, bank angle, metadata
│   ├── Spline                 # Catmull-Rom sampler → ordered centerline points
│   ├── PathSampler            # walks spline at fixed spacing; carries width/profile/bank per sample
│   ├── TerrainSampler         # reads surface height under samples via read-only WorldView
│   └── PathMask               # union of per-sample discs → occupancy field w/ soft edge distance
│
├── organic/                   # Subsystem 3: Beautification (PURE — the hard part)
│   ├── noise/                 # seeded Perlin + cellular/Worley, self-contained (no deps)
│   ├── EdgeShaper             # erode / grow / blur the mask edge
│   ├── GradientEngine         # weighted palette → clustered material assignment
│   ├── ProfileEngine          # cross-section (flat / crowned / ditch / raised)
│   ├── BlendEngine            # feather palette into surrounding terrain over an N-block skirt
│   └── BiomePresets           # biome → recommended palette
│
├── plan/                      # The spine
│   ├── EditPlan               # immutable sparse map: BlockPos → PlannedChange (place/remove/terrain)
│   └── EditPlanBuilder        # runs geometry→organic pipeline into an EditPlan (pure)
│
├── preview/                   # apply/render side
│   └── PreviewRenderer        # batched translucent mesh of the EditPlan, per chunk-section
│
├── apply/                     # apply/render side + the MP-ready boundary
│   ├── PlacementEngine        # applies/reverts an EditPlan against the server world
│   ├── EditHistory            # undo/redo stack of before-images
│   └── net/                   # ships an EditRequest to the server authority (thin in SP)
│
└── config/                    # palettes, biome presets, path styles, keybinds, tunables (JSON)
```

### 4.3 The three boundary rules

- **`geometry/` and `organic/` never import Minecraft world/render classes.** They take a `WorldView` read interface and plain data. This is what makes subsystems 2 and 3 headlessly testable and loader-portable.
- **`EditPlanBuilder` is the only code that runs the pipeline.** Preview and commit both consume its output; neither recomputes geometry. This operationally guarantees `preview ≡ commit`.
- **`apply/net/` is the server-authority seam.** Singleplayer = a thin local call. Dedicated multiplayer later = implement the packet handler behind the *same* `EditRequest` interface, no changes above it.

---

## 5. The `EditPlan` spine

`EditPlan` is the immutable contract between compute, preview, and apply.

- **Contents:** a sparse collection of `PlannedChange` keyed by `BlockPos`. Each `PlannedChange` carries the target `BlockState` and a **kind**: `PLACE`, `REMOVE`, or `TERRAIN` (used for preview coloring — green / red / blue respectively).
- **Immutable & pure-produced.** Built only by `EditPlanBuilder`; never mutated after construction. Recomputed (debounced) when control points or parameters change.
- **Consumers:**
  - `PreviewRenderer` — reads it to build the translucent mesh.
  - `PlacementEngine` — reads it to apply (and to capture before-images for undo).
- **Determinism guarantee:** because it is a pure function of `(points, params, seed)`, the plan shown in preview is the identical object applied on commit.

---

## 6. The pipeline (data flow)

`EditPlanBuilder` runs these pure stages in order; each is a data-in/data-out transform seeded off the single operation seed.

```
control points + params + seed
      │
      ▼
[Spline]         Catmull-Rom → smooth centerline (no jagged straight segments)
      │
      ▼
[PathSampler]    sample every ~0.5 blocks; carry width, profile, bank per sample
      │
      ▼
[TerrainSampler] read surface height under each sample (WorldView) → path hugs terrain
      │
      ▼
[PathMask]       union of per-sample discs → occupancy field with soft edge distance
      │
      ▼
[EdgeShaper]     Perlin + cellular noise → erode / grow / blur the border (irregular edges)
      │
      ▼
[ProfileEngine]  apply cross-section → per-column target heights (crown / ditch / raised)
      │
      ▼
[GradientEngine] weighted palette + clustered noise → material per block (clustered, not per-block random)
      │
      ▼
[BlendEngine]    feather materials into surrounding terrain over an N-block skirt
      │
      ▼
   EditPlan      immutable sparse set of PlannedChange
```

**Terrain-following & slope rules** (in `TerrainSampler` + `ProfileEngine`): slope below threshold → leave terrain; moderate → slight carve; steep → build stairs; extreme → suggest switchbacks (advisory, later milestone).

---

## 7. Cross-cutting mechanics

- **Undo/redo.** `PlacementEngine` captures a **before-image** of every block it touches into `EditHistory` at commit. Undo restores the before-image; **the operation is the atomic unit** (undo reverts the whole path, not point-by-point). Default depth ~20 operations, configurable. Before-images store only the affected region, not full chunks.
- **Preview rendering.** `PreviewRenderer` builds a **batched translucent mesh per chunk-section**, rebuilt only when the `EditPlan` changes (debounced ~100 ms after the last edit) — not N individual block renders. Green = place, red = remove, blue = terrain, via a custom translucent `RenderType`. Target: a ~5,000-block path preview at framerate.
- **Config / persistence.** Palettes, biome presets, path styles (M6), and tunables are JSON in the config dir, hot-reloadable. Keybinds via Forge `KeyMapping`.
- **Networking seam.** All mutation flows through `apply/net/` as an `EditRequest`. SP resolves it locally against the integrated server; MP later resolves it via a Forge `SimpleChannel` packet + server-side permission check. Nothing above the seam changes.

| 1. Interaction/UI | `tool/` | Straightforward |

The `organic/` engine — specifically `GradientEngine` + `EdgeShaper` — is where the project's value and risk concentrate.

## 9. Milestone roadmap

### Milestone 1 — Interaction Layer (**BUILD-READY**)

Delivers a working, visible tool loop with **zero world edits**.

**In scope:**
- Mod bootstrap: Forge 1.20.1 project, `@Mod` entry, client setup, `KeyMapping` registration.
- `ToolManager` + tool state machine (`Idle → Placing → Preview`); Alt+Scroll radial **tool wheel** (`ToolRenderer`); tool selection.
- `ToolInput`: right-click ray-casts to a `BlockPos` and appends a `PathNode`; undo/delete last point; Esc cancels.
- **`geometry/Spline` (Catmull-Rom) + `PathSampler`** — pure, unit-tested headlessly. (This one piece of Subsystem 2 is pulled forward so the interaction layer has a real curve to draw.)
- `ToolRenderer` draws control-point gizmos and the **line-strip centerline** of the spline in-world (debug render, not blocks).

**Explicitly out of scope for M1:** `EditPlan`, block preview mesh, any placement, width, organic stages, terrain modification.

**Definition of done:** with the Path tool selected via the wheel, place points in-world, watch a smooth Catmull-Rom curve update live, delete/cancel points — **and no blocks change.**

### Milestones 2–6 (design/interface level)

- **M2 — Path Geometry:** adjustable width (constant first) → `PathMask`; `TerrainSampler` surface-following; first real `EditPlan` produced.
- **M3 — Preview & Editing:** `PreviewRenderer` translucent block mesh; width handles; live update while dragging points; confirm/cancel workflow. First blocks placed on commit.
- **M4 — Terrain Modification:** replace top blocks only; preserve terrain shape; opt-in cut/fill grading; basic slope handling (stairs).
- **M5 — Organic Generation (the big one):** `EdgeShaper` (erode/grow/blur), `GradientEngine` (weighted palette + clustered noise), `BlendEngine` (edge feathering); exposed tunables (edge-randomness %, cluster size, per-block weights).
- **M6 — Advanced Builder Features:** variable width along path; cross-section profiles (`ProfileEngine`); biome-aware presets (`BiomePresets`); decorative pass (rocks/roots/foliage cleanup); save/load reusable path styles.

## 10. Primary engineering risks

1. **`GradientEngine` + `EdgeShaper` noise tuning (highest).** Choosing noise scales/thresholds that read as *intentional imperfection* rather than TV static or blobs is the core value and hardest tuning. Mitigation: build these with all tunables exposed early in M5, iterate against real screenshots, keep the stage pure so parameter sweeps are cheap and reproducible.
2. **Preview performance at large scale.** Rebuilding a big translucent mesh on every edit could stutter. Mitigation: per-chunk-section batching + debounced rebuilds + only re-mesh changed sections.
3. **Determinism leaks.** Any unseeded RNG or wall-clock read silently breaks `preview ≡ commit`. Mitigation: single operation seed threaded through the pipeline; a headless test asserts identical `EditPlan` across repeated builds.
4. **`WorldView` abstraction friction.** The read-only view must expose enough (heightmap, block states, biome) without leaking mutable world APIs into pure modules. Mitigation: define `WorldView` minimally and grow it deliberately.

---

## 11. Testing strategy

- **Pure modules (`geometry/`, `organic/`, `plan/`)** are unit-tested headlessly with a fake `WorldView`. Determinism test: build the same `EditPlan` twice, assert byte-identical.
- **Golden-image tests** for `organic/` stages: fixed seed + fixed inputs → snapshot the resulting material/edge grid; guard against regressions in the "looks hand-built" behavior.
- **`apply/` and `preview/`** verified in-game against M-level definitions of done.

---

## 12. Out of scope (this document / near term)

- Dedicated-multiplayer server component (designed-for, not built until after M5).
- Non-path tools beyond stubs (Line/Circle/Fill share the tool framework but are specified later).
- Custom schematic import in the decoration pass (M6 stretch).
- Any loader other than Forge 1.20.1 (the compute/apply split keeps this option open; it is not a near-term goal).
